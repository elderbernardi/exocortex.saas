# Issue Body Outline for Exocórtex Plans

Use this outline when converting an Exocórtex plan into GitHub issues.

## META issue outline

### Context
Why this initiative exists now.

### Objective
What the full issue graph should achieve.

### Strategy
Why the work is split in phases.

### Plan
- Phase 1
- Phase 2
- Phase 3

### Explicit dependencies
Spell out the graph in plain text.

### Definition of done
Checklist for closing the META issue.

### Notes for smaller agents
State the invariants they must preserve.

---

## Subissue outline

### Context
What is already true.

### Problem
What is wrong, missing, or ambiguous.

### Objective
What this issue must accomplish.

### Dependency
Blocked by / blocks.

### Scope
What belongs here.

### Non-scope
What explicitly does not belong here.

### Detailed tasks
Ordered, concrete steps.

### Expected deliverables
Files, docs, or behaviors that should exist after completion.

### Acceptance criteria
Binary-ish checks whenever possible.

### Notes for smaller agents
Warnings, invariants, routing rules, anti-patterns.

### Target files/areas
List the expected paths or surfaces.

### Related issues
Link the wider plan.

---

## Writing rules
- Write for a future agent with no chat context.
- Distinguish current state from target state.
- Prefer narrow executable scope over broad aspiration.
- Put ambiguity in the body now; do not leave it implicit.
- If the work is architectural, explain why the dependency exists, not only that it exists.
