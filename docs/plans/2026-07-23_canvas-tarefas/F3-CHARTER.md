# F3 — Sala viva conduzida (charter)

> **Charter, não plano.** Plano detalhado após gate F2.

## Objetivo

A sessão lançada atualiza o canvas em tempo real e é conduzida pelo loop fable-method (think/act/prove), com HITL restrito às 3 classes e bounds mecânicos.

## Entregáveis

1. **Assinatura de eventos da sessão → canvas**: artefatos produzidos viram cards (manifest EX-22); `clarify` do runtime vira gap card na fila; estado kanban sincronizado (EX-10); next_moves atualizados.
2. **Gates de governança visuais**: ação externa → cartão Draft-First com linha AUTH (palavras exatas do executivo registradas em `authorization[]` do canvas v0.5); INTENT/TWINS/PENDING do método → trace cards clicáveis (verificáveis, P9).
3. **Bounds como gatilhos de HITL**: 3 ciclos verify falhos → interrupt com hipótese e devolução; 2 buscas vazias → lacuna registrada; surprise (código×check×spec) → card de achado principal com ordem de autoridade executivo > spec > tests > código.
4. **Condução por fases do loop** (classify → define done → evidence → decide → act → verify → report) visível discretamente no canvas — nunca narrada no output do agente (regra do método).
5. **Decisão do port**: skills `excrtx-conduct-*` (compiled_rules → SOUL_SEED, EX-IDs + dogfood) OU metodologia embutida no prompt do enquadrador/sessão — executar o que a ADR da meta issue definir com evidência da F1/F2.

## Gate de saída

Execução real de uma tarefa não-trivial com: canvas refletindo a execução ao vivo; ≥1 cartão Draft-First com AUTH exercido; interrupções SOMENTE das 3 classes (log de interações anexado como prova).

## Guardrails específicos

Nunca enfraquecer os gates para "passar" (proibição do método); eventos novos entram na taxonomia do contrato SSE v1 (E9) com nomes estáveis documentados no contrato COLLAB.
